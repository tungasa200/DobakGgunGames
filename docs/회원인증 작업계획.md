## 현황 분석

### 현재 인증 구조

| 구성요소 | 현재 상태 |
|---|---|
| 사용자 식별 | IP 해싱 (SHA-256) |
| 점수 제출 검증 | GameSession (UUID) + IP Rate Limit |
| 회원 시스템 | **없음** (User Entity 미존재) |
| 인증 토큰 | **없음** (HMAC 레거시 필드만 존재, 미사용) |
| 소셜 로그인 | **없음** |

### 추가/변경이 필요한 것

- [ ] `User` Entity 및 Repository 신규 생성
- [ ] Spring Security 설정 (`SecurityFilterChain`)
- [ ] JWT 발급/검증 유틸리티
- [ ] Refresh Token Redis 저장소
- [ ] 이메일 인증 (Spring Mail + SMTP)
- [ ] OAuth2 소셜 로그인 (Google/Naver/Kakao)
- [ ] 프로필 사진 업로드 (Cloudflare R2 — SDK 이미 포함)
- [ ] 기존 `RankingRequest`의 레거시 `token`, `timestamp` 필드 제거

```
User
├── id           Long (PK, auto increment)
├── email        String (unique, not null)
```

```
EmailVerification

├── expiresAt LocalDateTime
└── used      boolean
```

### Step 4. Spring Security 설정

**파일**: `config/SecurityConfig.java`

> **원칙: 기존 서비스는 인증 없이 그대로 동작해야 한다.**  
> JwtAuthenticationFilter는 AT가 있으면 SecurityContext를 채워줄 뿐, 없어도 요청을 통과시킨다.

**접근 정책**

| 경로 | 정책 | 비고 |
|---|---|---|
| `/api/auth/**` | `permitAll` | 회원가입·로그인·토큰 재발급 등 |
| `/api/*/session/**` | `permitAll` | 기존 게임 세션 (변경 없음) |
| `/api/*/rankings` (GET/POST) | `permitAll` | 기존 랭킹 조회·제출 (변경 없음) |
| `/api/*/guess`, `/api/*/moves-batch` 등 | `permitAll` | 기존 게임플레이 엔드포인트 (변경 없음) |
| `/api/users/me/**` | `authenticated` | 신규 — 로그인 필수 |
| 그 외 `/api/**` | `permitAll` | 추후 개별 제한 가능 |

- `JwtAuthenticationFilter` — AT 헤더가 존재하고 유효하면 SecurityContext 주입, 없으면 그냥 통과
- CSRF 비활성화 (REST API)
- Session Stateless
- CORS는 기존 `WebConfig`에서 Spring Security CORS 설정으로 이전 (allowedHeaders에 `Authorization` 추가)

### Step 5. Auth API 구현

**파일**: `controller/AuthController.java`  
**파일**: `service/AuthService.java`

#### 5-1. 회원가입 `POST /api/auth/signup`

1. 이메일·닉네임 중복 검사
2. 닉네임 정책 검증 (2~12자, 욕설 필터)
3. BCrypt 비밀번호 해시
4. `User` 저장 (status=`PENDING`)
5. `EmailVerification` 토큰 생성 (UUID, 24h 만료) → DB 저장
6. 인증 이메일 발송

#### 5-2. 이메일 인증 `GET /api/auth/verify?token={token}`

1. 토큰 조회 → 만료/사용 여부 확인
2. `User.status` → `ACTIVE` 업데이트
3. `EmailVerification.used` → `true`

#### 5-3. 로그인 `POST /api/auth/login`

1. 이메일로 `User` 조회
2. `status == PENDING` 이면 "이메일 인증 필요" 응답
3. BCrypt 비밀번호 검증
4. AT + RT 발급
5. Redis에 RT 저장
6. 응답: `{ accessToken, refreshToken, user: { id, nickname, profileImage } }`

#### 5-4. 토큰 재발급 `POST /api/auth/refresh`

1. RT 서명/만료 검증
2. Redis에서 해당 userId RT와 일치 여부 확인
3. **RT Rotation**: 기존 Redis 키 삭제 → 새 AT + RT 발급 → 새 RT Redis 저장

#### 5-5. 로그아웃 `POST /api/auth/logout`

1. AT에서 userId 추출
2. Redis에서 `refresh_token:{userId}` 삭제

#### 5-6. 비밀번호 재설정 요청 `POST /api/auth/password-reset`

1. 이메일로 `User` 조회
2. 재설정 토큰 생성 (UUID, 30분 만료) → Redis 저장
3. 재설정 링크 이메일 발송

#### 5-7. 비밀번호 재설정 완료 `POST /api/auth/password-reset/confirm`

1. Redis에서 토큰 검증
2. 새 비밀번호 BCrypt 해시 → 저장
3. Redis 토큰 삭제 + 전체 세션 무효화 (`refresh_token:{userId}` 삭제)

### Step 6. 소셜 로그인 (OAuth2)

**파일**: `service/CustomOAuth2UserService.java`  
**파일**: `handler/OAuth2SuccessHandler.java`

#### 흐름

```
브라우저 → GET /oauth2/authorization/{provider}
         → Spring OAuth2 Provider 리다이렉트
         → 콜백: /login/oauth2/code/{provider}
         → CustomOAuth2UserService.loadUser()
         → OAuth2SuccessHandler.onAuthenticationSuccess()
         → AT + RT 발급 → 프론트엔드로 리다이렉트
```

#### `CustomOAuth2UserService` 처리 로직

1. Provider별 응답 파싱 (Google/Kakao/Naver 응답 구조 상이)
2. `email`로 기존 `User` 조회
   - 없으면: 자동 가입 (닉네임 자동 생성, status=`ACTIVE`)
   - 있고 `provider`가 다르면: 에러 또는 계정 연동 처리
3. `OAuth2SuccessHandler`에서 AT + RT 발급 후 프론트엔드 URL로 리다이렉트
   - 예: `https://도메인/oauth/callback?token={AT}`

## 기존 서비스 호환성 원칙

> 회원 인증 시스템은 **기존 게임 서비스에 영향을 주지 않는 방향**으로 구현한다.

| 항목 | 방침 |
|---|---|
| 게임 세션 (`/api/*/session/**`) | 인증 없이 동작 — 변경 없음 |
| 랭킹 조회/제출 (`/api/*/rankings`) | 인증 없이 동작 — 변경 없음 |
| 게임플레이 엔드포인트 (`/api/*/guess` 등) | 인증 없이 동작 — 변경 없음 |
| 랭킹 `userId` 연동 | 비로그인 → `null`, 로그인 → 자동 추출 (제출자가 의식하지 않아도 됨) |
| JwtFilter 동작 | AT 없으면 익명으로 통과, 있으면 SecurityContext 채움 |

## 보안 고려사항

| 항목 | 처리 방법 |
|---|---|
| AT 저장 위치 | `memory` (상태관리) — XSS 노출 최소화 |
| RT 저장 위치 | `httpOnly cookie` 권장 — JS 접근 불가 |
| RT Rotation | 재발급 시 기존 Redis 키 삭제, 새 키 저장 |
| 탈취된 RT 감지 | Redis에 없는 RT로 재발급 시도 시 → 전체 세션 무효화 |
| CORS | Spring Security CORS 설정으로 이전 (기존 `WebConfig` 통합) |
| CSRF | 비활성화 (Stateless REST API) |
| 비밀번호 재설정 토큰 | Redis 저장, TTL 30분, 1회 사용 후 삭제 |
