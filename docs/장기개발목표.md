# DobakGgun Games — 장기 개발 목표

> 현재 DobakGgun은 비회원 기반의 솔로 미니게임 서비스입니다.
> 이 문서는 **회원 시스템 → 실시간 멀티플레이 → 고성능 인프라** 순으로
> 서비스를 점진적으로 확장하기 위한 개발 로드맵입니다.

---

## Phase 1. 회원 인증 시스템

> 랭킹과 채팅 등 모든 소셜 기능의 전제 조건.
> 현재 HMAC 익명 점수 제출 방식을 대체하여 신뢰할 수 있는 사용자 식별 체계를 구축한다.

### 1-1. 회원가입

| 항목 | 내용 |
|---|---|
| 수집 정보 | 이메일, 닉네임, 비밀번호 |
| 비밀번호 저장 | BCrypt 단방향 해시 (Spring Security 기본 인코더) |
| 닉네임 정책 | 2~12자, 중복 불가, 욕설 필터 |
| 이메일 인증 | 가입 시 인증 메일 발송 → 링크 클릭 전까지 `PENDING` 상태 유지 |
| 프로필 사진 | 선택 항목, 가입 후 마이페이지에서 업로드 가능 |

**고려사항**
- 이메일 인증 토큰은 UUID + 만료 시간(24h)으로 생성, DB 또는 Redis에 임시 저장
- 인증 완료 전 로그인 시 "이메일 인증을 완료해주세요" 안내 응답

### 1-2. 프로필 사진 업로드

- 업로드 방식: 클라이언트 → 백엔드 → **S3 (또는 Cloudflare R2)** 저장
- DB에는 이미지 URL만 저장, 실제 파일은 오브젝트 스토리지에서 서빙
- 소셜 로그인 가입 시 소셜 프로필 이미지를 기본값으로 사용

| 항목 | 내용 |
|---|---|
| 허용 형식 | JPEG, PNG, WebP |
| 최대 용량 | 5MB |
| 리사이징 | 서버에서 256×256으로 리사이징 후 저장 (썸네일 통일) |
| 기본 이미지 | 프로필 미설정 시 닉네임 이니셜 기반 기본 아바타 표시 |

```
POST /api/users/me/profile-image    프로필 사진 업로드 (multipart/form-data)
DELETE /api/users/me/profile-image  프로필 사진 삭제 (기본 이미지로 복원)
```

**고려사항**
- 이전 사진 교체 시 S3에서 기존 파일 삭제하여 스토리지 낭비 방지
- 이미지 업로드는 인증된 사용자만 가능 (JWT 필수)

### 1-2. 로그인 및 토큰 발급

- **Access Token (JWT)**: 만료 15분, 사용자 ID·닉네임·권한 포함
- **Refresh Token**: 만료 7일, 서버 측(Redis)에 저장하여 무효화 가능
- 로그아웃 시 서버의 Refresh Token 즉시 삭제 → 강제 만료

```
POST /api/auth/signup       회원가입
POST /api/auth/login        로그인 → AT + RT 발급
POST /api/auth/refresh      AT 재발급 (RT 검증)
POST /api/auth/logout       RT 삭제
GET  /api/auth/verify?token 이메일 인증
```

### 1-3. OAuth 소셜 로그인

지원 플랫폼: **Google**, **Naver**, **Kakao**

- Spring Security OAuth2 Client 사용
- 소셜 로그인 시 최초 진입이면 자동 가입 (닉네임은 소셜 프로필 이름 기반 자동 생성)
- 이미 가입된 이메일로 다른 소셜 로그인 시도 시 계정 연동 또는 에러 처리

### 1-4. 보안 강화 포인트

- Refresh Token Rotation: RT 재발급 시 기존 RT 폐기, 새 RT 발급
- 비정상 Refresh Token 사용 감지 시 해당 계정 전체 세션 무효화
- 비밀번호 재설정: 이메일 인증 링크 방식 (토큰 만료 30분)

## Phase 2. 실시간 채팅

> 게임 로비 또는 전용 채팅방에서 로그인 사용자끼리 실시간으로 소통하는 기능.
> 채팅 내용은 영속적으로 저장하지 않고 Redis 기반으로 휘발성으로 운영한다.

### 2-1. WebSocket 서버 구성

- **프레임워크**: Spring WebSocket + STOMP 프로토콜
- **엔드포인트**: `/ws` (SockJS 폴백 지원)
- STOMP 기반 pub/sub 구조로 채팅방별 메시지 라우팅

```
SUBSCRIBE  /topic/room/{roomId}     채팅방 메시지 수신
SEND       /app/chat/{roomId}       메시지 발신
SUBSCRIBE  /queue/errors            개인 에러 수신
```

### 2-2. 인증 연동

- WebSocket 핸드셰이크 단계에서 JWT 검증
- 인증되지 않은 사용자는 연결 자체를 거부 (401)
- 채팅 메시지에 서버가 발신자 닉네임 주입 → 클라이언트 위조 불가

### 2-3. 채팅방 설계

| 유형 | 설명 |
|---|---|
| 글로벌 채팅 | 서비스 전체 공개 채팅방 (1개 고정) |
| 게임 로비 채팅 | 게임별 채팅방 (예: `/topic/room/minesweeper`) |

- 채팅 메시지는 Redis List(`LPUSH` + `LTRIM`)에 최근 100개만 보관
- 채팅방 연결 시 Redis에서 최근 메시지 히스토리 제공 (페이지 로드 효과)
- 채팅방이 비워지면(마지막 사용자 퇴장) Redis 키 자동 만료 설정

## Phase 3. Redis 인프라 통합

> 인증 토큰 관리, 채팅 데이터, 실시간 랭킹 캐싱 등 여러 기능을 Redis로 통합하여
> DB 부하를 줄이고 응답 속도를 개선한다.

### 3-1. Refresh Token 저장소

```
Key:   refresh_token:{userId}
Value: {token}
TTL:   7일
```

- 로그아웃 또는 토큰 로테이션 시 즉시 `DEL`
- RT Rotation 구현: 재발급 요청마다 기존 키 삭제 + 새 키 저장

### 3-2. 채팅 메시지 캐시

```
Key:   chat:room:{roomId}
Type:  List (LPUSH + LTRIM 100)
TTL:   채팅방 비활성 1시간 후 자동 만료
```

- 메시지 구조: `{ userId, nickname, message, timestamp }`
- 채팅방 입장 시 `LRANGE chat:room:{roomId} 0 49` 로 최근 50개 히스토리 제공

### 3-3. 랭킹 캐시 (선택적 확장)

현재 랭킹은 매 요청마다 DB 쿼리 발생 → 트래픽 증가 시 병목 가능성.

```
Key:   ranking:{game}:{level}:weekly
Value: JSON (TOP 10 직렬화)
TTL:   5분 (자동 갱신)
```

- 점수 제출 시 해당 게임·레벨의 캐시 키 `DEL` → 다음 조회 시 DB에서 재적재
- 또는 Redis Sorted Set(`ZADD`)으로 실시간 랭킹 직접 관리 (DB 부하 최소화)

## 전체 로드맵 요약

```
Phase 1 — 회원 인증       ████████░░░░  진행 예정
Phase 2 — 실시간 채팅     ░░░░░░░░░░░░  Phase 1 완료 후
Phase 3 — Redis 통합      ░░░░░░░░░░░░  Phase 1~2 병행
```

| Phase | 주요 의존성 | 완료 기준 |
|---|---|---|
| 1. 인증 | - | JWT 로그인·소셜 로그인·이메일 인증 동작 |
| 2. 채팅 | Phase 1 (JWT) | 인증 사용자 간 실시간 메시지 송수신 |
| 3. Redis | Phase 1 (RT), Phase 2 (채팅) | RT·채팅 데이터 Redis 저장, 로컬 캐시 제거 |

## 기술 스택 변경 사항

| 항목 | 현재 | 추가 예정 |
|---|---|---|
| 인증 | HMAC 익명 서명 | Spring Security + JWT |
| 소셜 로그인 | 없음 | OAuth2 (Google/Naver/Kakao) |
| 실시간 통신 | 없음 | Spring WebSocket + STOMP |
| 캐시/세션 | 없음 | Redis (Lettuce 클라이언트) |
| 메일 발송 | 없음 | Spring Mail (SMTP) |
| 파일 스토리지 | 없음 | S3 / Cloudflare R2 (프로필 사진) |
