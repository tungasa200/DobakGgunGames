## A. 랭킹 테이블 모바일 가로 오버플로우 수정 (`797c20d`)

### 문제
가로 모바일 뷰포트에서 긴 닉네임이 등록된 경우, 랭킹 테이블 너비가 디바이스 폭을 넘어 페이지 가로 스크롤이 발생.

### 원인
- **Apple, Blockfall(일반)**: `.rankTable`/`.table` 전체에 `white-space: nowrap` 적용 — 긴 닉네임이 셀을 강제 확장
- **Baseball, Sudoku, Minesweeper, Solitaire**: nowrap은 없지만 `word-break` 미설정 — 긴 영문 단일 토큰이 min-content를 키워 테이블이 컨테이너 초과

### 수정
6개 게임 동일 패턴 적용:
1. 테이블 단위 `nowrap` 제거 → `td` 단위로 이동 (날짜·점수 열은 한 줄 유지)
2. 이름 열(`:nth-child(2)`)에만:
   ```css
   white-space: normal;
   word-break: break-word;
   overflow-wrap: anywhere;
   ```

### 손대지 않은 부분
- **BlockfallInsane**: 이미 `table-layout: fixed` + ellipsis로 처리되어 있음
- **Excel 모드 grids**: grid 고정 폭 + `overflow: hidden`으로 잘림 — 문제 없음

## B. 블록폴 게임 영역 + 랭킹 폭 통일 + 셀 확대 (`916609f`, `770d626`)

### 변경 의도
- 인세인 모드 랭킹이 게임 영역(598px)보다 좁게(560px) 보이는 misalignment 해소
- 데스크탑에서 게임 보드를 적절히 확대해 가시성 향상

### CELL 상수 변화
- 30 → 40 (시도) → **33** (최종)
- 40은 너무 커서 33으로 안착 (1.1배 확대)
- `ctx.scale(CELL, CELL)` / `setTransform` 기반이라 상수만 바꿔도 보드·NEXT·HOLD·BAG 캔버스가 균일하게 확대됨

### 사이드 패널 일괄 132px (= 4 × CELL)
| 항목 | 이전 (CELL=30) | 변경 (CELL=33) |
|---|---|---|
| miniCanvas | 120×120 | **132×132** |
| sidePanel | 120 | **132** |
| bagPanel | 120 | **132** |
| bagPanelCanvas | 120×630 | **132×693** |

### 컨테이너 폭
| 항목 | 이전 | 변경 |
|---|---|---|
| `.wrap` max-width | 640px | **687px** (655 + 패딩 32) |
| `.playWrapper` width | 594px | **655px** (gameArea 실측과 일치) |
| 인세인 `.rankSection` max-width | 560px | **655px** (일반과 폭 통일) |

### 미디어 쿼리 ≤686px (반응형 롤백)
뷰포트 < 687px 시 데스크탑 확대를 기존 120/330 크기로 CSS 롤백:
```css
@media (max-width: 686px) {
  .playWrapper { width: auto; }
  .bagPanel { display: none; }
  .sidePanel { width: 120px; }
  .miniCanvas { width: 120px; height: 120px; }
  .bagPanelCanvas { width: 120px; height: 630px; }
  .board { width: 330px; height: 630px; }
}
```
캔버스 내부 해상도는 CELL=33로 그대로 유지되어 모바일에서 더 선명한 렌더링.

## C. 인세인 BAG UI 레이아웃 안정화 (`f185fdc`)

### 문제
인세인 페이즈 진입 후 7-bag 소진 시 `bagVisible=false`로 BAG 패널이 DOM에서 제거되어 gameArea 너비가 655 → 511px로 줄어 아래쪽 rankSection(655px)과 misalign.

또한 페이지 컨텐츠 높이 변화(랭킹 vs 룰 탭, 모달 등)에 따라 세로 스크롤바가 토글되며 뷰포트 가용 너비가 ~15px 출렁이는 부수 문제.

### 수정
1. `bagPanel` 래퍼 `<div>`는 항상 렌더 → 132px 레이아웃 공간 유지
2. 내부 `sideBox`(빨간 박스 + BAG 라벨 + 캔버스)만 `bagVisible` 조건부 렌더
3. `index.css`에 `html { scrollbar-gutter: stable; }` 추가 → 세로 스크롤바 공간 항상 예약

## D. 콤보/보너스 텍스트 HTML 오버레이화 (`700e52c`)

### 문제
canvas 내부에 `strokeText/fillText`로 그리던 콤보/보너스 텍스트가 보드 폭(11×CELL=363px)보다 길면 좌우가 잘림. `COMBO x3 +500,000` 같은 긴 메시지가 끝까지 안 보임.

### 수정
- canvas 렌더 블록 제거 → HTML `div` 오버레이로 이동 (PAUSE 오버레이와 동일 패턴)
- React state `comboOverlay: { text, key }` + `comboOverlayKey` ref로 관리
- `boardWrapper` 내부 `position: absolute`로 캔버스 위에 띄움
- CSS `comboOverlayPop` 애니메이션 (1.2s): 0.6x → 1.15x bounce → 1x → fade out
- 색상: Normal `#FFE138`, Insane `#ffd60a`
- 검정 4방향 `text-shadow`로 캔버스 strokeText 아웃라인 재현

### 영향
- 텍스트가 캔버스 밖으로 자유롭게 뻗어나가 잘리지 않음
- 스타일은 기존 노란 글자 + 검정 아웃라인 그대로 유지

## F. 측면 붕괴(SIDE_EXPAND) 이벤트 신설 (`fd01323`)

### 컨셉
인세인 페이즈 7-bag 소진 후 비어있는 BAG 패널 공간을 활용해 보드 폭을 확장하는 새 이벤트. 바닥 붕괴(FLOOR_DROP)의 수평 버전.

### 메커닉
- **이벤트**: `🧱 측면 붕괴`, type=physical, mobileExcluded, HIGH_FLASH 등급
- **토글 방식**: 11 ↔ 15
  - 기본 폭(11)에서 발동 → +4 확장 (11 → 15)
  - 이미 확장(15) 상태에서 발동 → 원복 (15 → 11)
- **발동 조건**: `bagVisibleRef.current === false` (인세인 페이즈 + bag 소진 후)

### 레이아웃 처리
- `boardExpanded` state로 `.bagPanel` 래퍼 자체 조건부 제거
- 확장된 boardBox(132+12=144px)가 bag 자리 흡수
- gameArea 폭은 655px 이내 유지

### 연출
- 모든 고정 블록을 shatter 파편으로 박살 (FLOOR_DROP과 동일 처리)
- triggerShake + hue-rotate(270deg) 보드 필터
- 0.2초 후 추가 흔들림(18, 800ms)

### 원복 시 처리
- 우측 초과 영역의 정착 파티클 제거
- 플레이어 피스 위치를 `boardW - pw`로 클램프 (게임오버 방지)

## H. 이벤트 지속시간 레벨 스케일링

### 적용
지속형 이벤트(FLIP_H/V, DARK_SPOTLIGHT, INVISIBLE_PIECE, COLOR_GRAY, CONTROL_FREEZE)의 지속시간을 Lv4부터 점진 감소.

```ts
const durMul = lv < 4 ? 1 : Math.max(0.2, 1 - (lv - 3) * 0.1);
```

| 레벨 | 배율 | FLIP_H(8s) | DARK(10s) | INVISIBLE(6s) |
|---|---|---|---|---|
| 1-3 | 100% | 8.0s | 10.0s | 6.0s |
| 4 | 90% | 7.2s | 9.0s | 5.4s |
| 7 | 60% | 4.8s | 6.0s | 3.6s |
| 9 | 40% | 3.2s | 4.0s | 2.4s |
| 11 | **20%(최소)** | 1.6s | 2.0s | 1.2s |

### 구현 포인트
- `activeEventDurInit` ref 신설 → 타이머바 비율 계산 분모로 사용 (스케일된 초기값 보존)
- SPIN_BLOCK(`duration: -1`)은 무관 (무제한 이벤트)
- `clearActiveEvent`에서 `activeEventDurInit.current = 0` 도 함께 초기화

## I. 점수 수급 보강 (Feature 1 + 3)

### Feature 1: CLEANSE 보너스
처치 불가 요소(DEAD_COLOR 블록 + DEAD 샌드 파티클)를 라인 클리어/폭발로 치웠을 때 보너스.

| 트리거 | 보너스 |
|---|---|
| 라인 클리어 인접 행의 DEAD 제거 | +30 × 레벨 × 개수 |
| 폭발 반경 내 DEAD 제거 | +30 × 레벨 × 개수 |

오버레이: `CLEANSE x{count} +{bonus}` (alpha 2.0)

### Feature 3: EVENT SURVIVED 보너스
지속형 이벤트가 자연 종료까지 버텨졌을 때 추가 점수.

```ts
survivalBonus = 50 × Math.max(1, round(scaledDur / 1000)) × level
```

예시:
- Lv1 FLIP_H(8초): 50 × 8 × 1 = **400점**
- Lv5 DARK(8초 스케일): 50 × 8 × 5 = **2,000점**
- Lv11 DARK(2초 스케일): 50 × 2 × 11 = **1,100점**

오버레이는 **주석 처리** (`5a682bc`) — 점수만 조용히 지급. 다른 오버레이(CLEANSE/COMBO)와 겹쳐 표시되는 빈도가 높아 시각적 피로 완화.

### 미적용
- **Feature 5 (오염도 배수)**: 사용자 요청으로 도입 후 철회

## J. 레벨링 완화 (정화 → 라인 크레딧)

### 동기
Feature 1+3로 점수만 부풀고 레벨링 자체 난이도(100라인 클리어)는 그대로 → 실제 어려움(줄 클리어)은 같은데 보상만 늘어난 상태.

### 메커닉
정화 행위가 레벨 진행에도 기여:
```ts
applyCleanseLineCredit(cleansedCount: number) {
  cleanseLineAccum.current += cleansedCount;
  while (cleanseLineAccum.current >= 5) {
    cleanseLineAccum.current -= 5;
    linesRef.current += 1;
  }
}
```

5개 정화 = 1 라인 크레딧. CLEANSE 보너스 발생 지점(라인 클리어/폭발)에서 호출.

### 레벨업 헬퍼 분리
기존 인라인 레벨업 로직을 `checkLevelUp()` 헬퍼로 추출:
- 라인 클리어 경로(arenaSweepInsane)와 정화 경로(EXPLODE) 양쪽에서 호출
- Lv9 invert, Lv11 cosmic horror 진입 효과를 일관 처리

### 효과
- Lv1→11 도달 ~10% 빠르게 (100라인 → ~88-94라인)
- 이벤트 조우 횟수도 ~10% 감소 → Feature 1+3 보너스 누적도 함께 감소
- 결과: 기존 90-100k 점수대 유지하면서 플레이 경험만 변화
