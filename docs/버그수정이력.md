# 버그 수정 이력

> 운영 중 발견된 버그를 아래에 계속 추가. 버그 발견 시 원인·수정 내용·수정 파일을 기록할 것.

## 수칙

1. **원인 먼저 파악** — 코드를 읽고 실제 원인을 확인한 뒤 최소 범위 수정. 관련 없는 코드는 건드리지 않을 것
2. **버그별 독립 수정** — 버그마다 영향 범위가 다르므로 한 버그 수정이 다른 버그에 영향을 주지 않도록 할 것
3. **원본 동작 기준** — 수정 후 원본(`excel-games/`)과 동일하게 동작하거나, 원본에 없는 버그라면 올바른 UX로 수정할 것

## ✅ 15: 엑셀모드 하단 플러스 버튼 클릭 시 새 게임 미시작

- Apple → `handleStart()`, Baseball → `reset(level); setInput(''); setHint(''); initSession(level)`, Blockfall → `startGame()`, Minesweeper → `reset(level)`, Solitaire → `handleStartGame(drawMode)`

## ✅ 5: 숫자야구 게임오버 없음

**원인**: `useBaseballGame.ts`의 `guess()` 함수가 정답(3S) 체크만 하고 `MAX_ATTEMPTS` 초과 체크를 하지 않음.
`State`에 `gameOver` 필드가 없고 reducer에도 `GAME_OVER` 액션이 없어 최대 횟수를 채워도 게임이 종료되지 않았음.

**수정**:

- `BaseballBoard.tsx` 입력 필드·확인 버튼에 `state.gameOver` 비활성화 조건 추가
- 게임오버 시 `"게임 오버! 정답은 XXXX 였습니다."` 힌트 표시

**수정 파일**: `games/baseball/useBaseballGame.ts`, `games/baseball/BaseballBoard.tsx`

## ✅ 6: 숫자야구 히스토리 테이블 빈 행 key 충돌

**원인**: 테이블 tbody 렌더링에서 데이터 행은 `key={row.attempt}`, 빈 행은 `key={i}`(0-based 인덱스)를 사용.
`row.attempt`와 `i`가 같은 숫자가 되는 경우(예: attempt=5, i=5) React key 충돌 발생.

**수정**: 빈 행 key를 `key={i}` → `` key={`empty-${rowNum}`} ``로 변경하여 데이터 행과 충돌 방지.

**수정 파일**: `games/baseball/BaseballBoard.tsx`

## ✅ 7: 숫자야구 정답 항상 `012` 고정

**원인**: `generateAnswer()`에서 `pool.sort(() => Math.random() - 0.5)` 방식 사용.
이 방식은 V8 엔진의 TimSort에서 비일관적 비교 함수를 전달하면 원래 순서(`[0,1,2,...]`)가 보존되는 편향이 발생함.

**수정**: Fisher-Yates 셔플로 교체.
```ts
for (let i = pool.length - 1; i > 0; i--) {
  const j = Math.floor(Math.random() * (i + 1));
  [pool[i], pool[j]] = [pool[j], pool[i]];
}
```

**수정 파일**: `games/baseball/useBaseballGame.ts`

## ✅ 8: 숫자야구 클리어 시간 17165469878초 출력

**원인**: `startTimeRef.current = 0`(초기값)인 상태에서 첫 번째 guess 시 GUESS + WIN dispatch가
React batching으로 같은 렌더에서 처리됨. `timerRunning: false→true→false`가 한 렌더에 묶여
`useEffect([timerRunning])`이 실행되지 않아 `startTimeRef`가 0으로 유지됨.
결과적으로 `elapsed = (Date.now() - 0) / 1000 ≈ Unix timestamp(초)` 값이 그대로 노출.

**수정**:
- `guess()` 함수 내에서 `startTimeRef.current === 0`이면 `Date.now()`로 직접 설정 (useEffect 의존 제거)
- `reset()` 호출 시 `startTimeRef.current = 0`으로 초기화

**수정 파일**: `games/baseball/useBaseballGame.ts`

**수정 파일**: `games/baseball/BaseballBoard.tsx`

## ✅ 10: 솔리테어 플레이타임 버그 (타이머 미동작 / 0초 기록)

**원인**: `useSolitaireGame.ts`의 `SET_GAME` 리듀서가 카드 첫 선택(클릭) 시 `status: 'idle' → 'playing'`으로 변경함.
이후 실제 이동 시 `ensureTimer`가 `state.status === 'idle'` 조건을 검사하는데 이미 `'playing'`이므로
`START_TIMER`가 dispatch되지 않아 `timerRunning`이 `false`로 유지 → 타이머 미동작 → `elapsed = 0`.

**수정**: `ensureTimer`의 조건을 `state.status === 'idle'` → `!state.timerRunning && state.status !== 'won'`으로 변경.
`status` 대신 `timerRunning` 플래그를 직접 확인.

**수정 파일**: `games/solitaire/useSolitaireGame.ts`

## ✅ 12: 블록폴 기록 등록 HTTP 400

**원인**: `BlockfallValidationService`의 점수 상한 계산이 `linesCleared × 800 × gameLevel × 3`으로만 산정됨.
소프트·하드 드롭 보너스(soft: +1/행, hard: +2/행)는 `linesCleared`와 무관하게 발생하므로
`linesCleared = 0`이어도 `score > 0`이 될 수 있음 → `score > maxScore(=0)` → BAD_REQUEST.

**수정**: `maxScore` 계산에 드롭 보너스 최대치 `MAX_DROP_BONUS = 400` (20행 × 10열 × 2점) 추가.
`maxScore = linesCleared × 800 × gameLevel × 3 + 400`

**수정 파일**: `backend/.../service/BlockfallValidationService.java`

## ✅ 13: 지뢰찾기 최초 클릭 딜레이 로딩 표시 없음
