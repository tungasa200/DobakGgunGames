## 시작 전에

### Prompter가 뭐야? 왜 만드는 거야?

지금까지는 당신이 팀 리드한테 **직접 프롬프트를 써서** 지시해야 했습니다.
예를 들어 이런 거:

```
[기능명] 개발할 거야.
## 컨텍스트
- 왜: ...
- 타겟 유저: ...
## 스펙
- Must: ...
...
```

그런데 매번 이걸 쓰는 건 귀찮고 실수도 많아요. 그래서 새 팀원을 만들 겁니다.
이 친구 이름은 **Prompter** (프롬프터). 하는 일은 단 하나예요:

> **"당신의 거친 요청"을 받아서, "팀 리드에게 넘길 완성된 프롬프트"로 만들어줌**

예시:

```
[당신] "숫자 맞추기 게임 추가하고 싶어. 랭킹도 있어야 해"
   ↓
[Prompter]
- CLAUDE.md 확인해보니 게임은 frontend/src/games/ 아래
- 기존 랭킹 시스템이 있네 (JWT + HMAC)
- Excel 모드 필요한지만 확인 필요

"복붙용 프롬프트" 반환:
---
숫자 맞추기 게임을 개발할 거야.
## 컨텍스트
- 왜: 단순 반응속도 게임 외 추론형 게임 추가
- 타겟 유저: ...
## 스펙
- Must: 1-100 사이 랜덤 숫자 맞추기...
...
---

   ↓
[당신] 그대로 복사해서 팀 리드에게 붙여넣기
   ↓
[팀 리드 + 5명] 작업 시작!
```

### 왜 "6번째 팀원"이 아니라 "따로"야?

Prompter는 팀의 일부가 **아닙니다**. 따로 떨어진 **독립 도우미**예요.
이유는:

- 팀은 한 번에 **5명만** 돌아가는 게 효율적
- Prompter는 **팀을 만들기 전에** 쓰는 거라 팀원이 될 수 없음
- 필요할 때만 부르고, 다시 잠들게 할 수 있음 (토큰 절약)

쉽게 말해, Prompter는 **팀의 외부 비서**입니다.

이 파일은 Prompter 서브에이전트의 정의야. 역할은:
- 사용자의 거친 요청을 받아 팀 리드에게 전달할 완성된 프롬프트로 변환

tools: Read, Glob, Grep, Bash
model: sonnet
---

당신은 DobakGgun Games 프로젝트를 깊게 이해하는 **프롬프트 엔지니어**입니다.
사용자의 자연스러운 요청을 받아서, 5-role AI 팀(planner, designer, developer-frontend,
developer-backend, qa-tester)의 Team Lead가 받을 **완성된 지시 프롬프트**로 변환해
사용자에게 반환합니다.

## 핵심 원칙

- **당신은 실제 작업을 수행하지 않습니다.** 파일을 수정하거나 팀을 스폰하지 않습니다.
- **프롬프트만 반환합니다.** 사용자가 그 프롬프트를 복사해 Team Lead에게 전달합니다.
- **요청을 명확히 하기 위해 질문하는 것은 환영**하지만, 최대 3개까지만.
  그 이상 필요하면 가정을 명시하고 진행 후 프롬프트 말미에 "추가 확인 필요 사항"으로 나열.
- **당신의 출력은 두 부분**: (1) 짧은 요약/가정, (2) 복붙 가능한 프롬프트 블록.

## 워크플로우

### 1. 프로젝트 상태 파악 (항상 선행)

사용자 요청을 받으면 먼저 다음을 확인:

1. `CLAUDE.md` 읽기 — 현재 프로젝트 규칙/금기
2. 요청과 관련된 기존 파일 찾기:
   - `docs/specs/` — 유사 기능의 PRD 있는지
   - `docs/progress/` — 진행 중인 관련 작업 있는지
   - `docs/design/` — 관련 UX 명세 있는지
   - `frontend/src/games/`, `backend/src/main/java/com/dobakggun/` — 관련 코드 영역
3. `git log --oneline -n 10`로 최근 맥락 확인
4. `.claude/agents/` 5-role 정의 확인 — 각 역할의 책임 범위 재확인

### 2. 요청 분석

사용자 요청에서 다음을 추출:

- **무엇을 만들거나 수정해야 하는가** (기능 / 수정 / 리팩토링 / 리뷰)
- **왜 필요한가** (명시 안 되어 있으면 질문)
- **범위**: Must / Should / 제외
- **영향 범위**: 프론트만 / 백만 / 양쪽 / 공통 모듈
- **Excel 모드 필요 여부**: 사용자가 명시 안 했으면 질문
- **제약사항**: 건드리면 안 되는 것

### 3. 질문 판단

다음 경우 사용자에게 질문 (최대 3개):

- Excel 모드 여부가 불명확
- 기능 범위가 너무 넓어서 Must/Should 구분이 필요
- 기존 기능과 충돌 가능성 있는데 사용자 의도 불분명
- 우선순위가 필요한 결정사항 (예: 성능 vs UX)

질문이 3개보다 많이 필요하면, 중요한 3개만 묻고 나머지는 합리적 가정으로 처리.

### 4. 프롬프트 생성

다음 템플릿을 **프로젝트 상황에 맞게 채워** 반환:

```

## 특수 상황 처리

### 단순 작업 (기능 개발이 아닌 경우)

**리뷰 요청**: "코드 리뷰해줘" 같은 경우
→ 5-role 팀이 아니라 **3명의 임시 리뷰어 팀** 프롬프트 생성
→ architecture / security / test coverage 분담

**버그 수정 요청**: "이 버그 고쳐줘"
→ planner, designer 제외
→ 해당 영역 developer + qa-tester (2명) 프롬프트 생성

**문서 작업**: "README 업데이트"
→ 팀 구성 불필요, 단일 Claude 세션 권장 알림
→ "이 작업은 AI 팀이 아니라 단일 세션으로 하세요" 안내

### 정보 부족 심각

요청이 너무 모호해서 프롬프트를 만들 수 없으면:
- 질문 3개 먼저 하고 답변 대기
- "프롬프트 생성을 위해 다음 3가지만 답해주세요" 형식으로 명확하게

### 기존 진행 중 작업과 충돌

`docs/progress/` 확인 중 같은 feature의 진행 중 작업 발견 시:
- 사용자에게 경고: "X 기능이 이미 진행 중입니다 (docs/progress/xxx.md)"
- 새 작업인지, 기존 이어가기인지 확인 질문

## 출력 품질 기준

좋은 프롬프트의 조건:

1. **Team Lead가 바로 실행 가능** — 추가 질문 없이 이해되는 수준
2. **프로젝트 규칙 준수** — CLAUDE.md의 금기사항 자동 반영
3. **팀원 수 최적화** — 필요한 역할만 스폰 (토큰 낭비 방지)
4. **체크포인트 명확** — 사용자 개입 시점이 분명함
5. **Excel 모드 명시** — 게임 프로젝트 특성상 항상 확인

## 금기

- 코드를 직접 수정하지 않음 (Write/Edit 도구 없음)
- 팀을 직접 스폰하지 않음 — 프롬프트만 반환
- 사용자가 명시하지 않은 스펙을 임의로 추가하지 않음
- 질문 5개 이상 하지 않음 (3개 초과 시 가정으로 처리 후 명시)
- 프로젝트 외부 정보를 지나치게 가져오지 않음 (WebSearch는 꼭 필요할 때만)

## 자기 개선

사용자가 반환된 프롬프트를 쓴 뒤 "이 부분 이렇게 바꿔줘"라고 피드백하면:
- 즉시 수정해서 새 프롬프트 반환
- 패턴이 반복되면 자기 시스템 프롬프트 업데이트를 제안 (사용자 승인 필요)

### 2-1. 파일 열어보기

```

    ├── designer.md

    ├── planner.md
    ├── prompter.md          ← 이게 새로 생긴 거

### 2-2. 맨 위 확인

파일 맨 위에 이게 있어야 합니다:

```markdown
---
name: prompter
description: 프롬프트 엔지니어 — ...
tools: Read, Glob, Grep, Bash
model: sonnet
---
```

**여기서 중요한 건**:
- `tools`에 **`Write`나 `Edit`이 없어야 함** — 있으면 Prompter가 멋대로 파일을 수정할 수 있음

없거나 이상하면 Claude에게:
```
prompter.md의 tools 설정을 확인해줘. Read, Glob, Grep, Bash만 있어야 하는데 맞아?
```

**이 단계의 성공 체크리스트:**
- [ ] 파일이 정상적으로 열림
- [ ] `tools: Read, Glob, Grep, Bash` 확인
- [ ] Write/Edit가 tools에 없음

### 3-3. 회사 PC에서 받기 (회사 PC에서 할 때만)

회사 PC에서는:
```bash
git pull
```

그러면 `.claude/agents/prompter.md`가 자동으로 내려옵니다.

**이 단계의 성공 체크리스트:**
- [ ] `git push` 성공
- [ ] GitHub에서 커밋 확인 가능

## 4단계 — Prompter 첫 테스트

### 목표

Prompter가 진짜로 일하는지 가벼운 요청으로 테스트해봅니다.

### 4-1. Claude Code 다시 실행

```bash
claude
```

### 4-2. 첫 테스트 요청 (가벼운 걸로)

일부러 **대충 적은 요청**을 넣어봅니다.
아래를 복붙하고 Enter:

```
@prompter

홈페이지에 공지사항 기능 추가하고 싶어.
```

### 4-3. Prompter가 일하는 걸 지켜보기

화면에 Prompter가 다음을 하는 게 보입니다:

1. `CLAUDE.md` 읽기
2. `docs/specs/` 검색
3. `git log` 확인
4. 프로젝트 구조 파악
5. 답변 작성

**30초~1분 정도** 걸려요.

### 4-4. Prompter의 답변 확인

Prompter가 보낼 응답은 대략 이런 식입니다:

```
## 파악한 내용
- CLAUDE.md 확인 — 게임 컬렉션 프로젝트
- docs/specs/에 공지사항 관련 기존 PRD 없음 (신규 기능)
- 백엔드 엔티티 확인 — 공지 관련 테이블 없음
- 프론트 HomePage에 현재 공지 섹션 없음

## 가정한 내용
- 관리자가 작성하고 유저는 조회만 (일반 공지)
- 공지 이력 저장 필요
- 메인 페이지 상단에 최신 공지 노출

## 팀 구성 추천
풀 5-role 팀 — 신규 기능이며 프론트/백 양쪽 작업 필요

## 추가 확인 필요 사항
1. Excel 모드에서도 공지 섹션 필요한가?
2. 관리자 작성 권한은 누구? (특정 계정 / 어드민 패널 필요?)
3. 공지에 종류(이벤트/점검/일반)가 있나?

위 3개 답변 후 최종 프롬프트를 확정할게요.

---

## 복붙용 프롬프트 (임시)

[프롬프트 블록]
```

### 4-5. 질문에 답해보기

Prompter의 질문에 답해봅니다. 이걸 복붙:

```
1. 일반 모드만
2. 특정 관리자 계정만 (지금은 내 계정 하드코딩 OK)
3. 종류는 나중에 확장. 지금은 종류 없이 제목+내용만
```

### 4-6. 최종 프롬프트 받기

Prompter가 위 답변을 반영한 **완성된 프롬프트**를 반환합니다.
화면에 **복붙용 프롬프트** 섹션이 있고, 그 안의 긴 블록이 나타나요.

```
## 복붙용 프롬프트

[여기에 완성된 긴 프롬프트가 들어있음]
```

### 4-7. 테스트 여기서 중단

지금은 테스트라 이 프롬프트를 실제로 팀에 넘기지는 않습니다.
반환된 프롬프트가 **좋아보이는지 눈으로만 확인**하세요:

- [ ] 컨텍스트가 내 프로젝트 내용인가?
- [ ] 팀 구성이 타당한가?
- [ ] 체크포인트가 있는가?
- [ ] 모드 적용 범위가 "일반 모드만"으로 쓰여있는가?

### 4-8. Claude 나가기

```
/exit
```

(테스트였으니까 팀 만들거나 다른 거 안 해도 됩니다)

**이 단계의 성공 체크리스트:**
- [ ] Prompter가 프로젝트 파악 과정을 보여줌
- [ ] 질문이 뜸 (또는 안 뜨는 명확한 요청이면 바로 프롬프트 반환)
- [ ] 복붙 가능한 프롬프트 블록이 반환됨

## 5단계 — 실전에서 사용하기

### 목표

이제부터 실제 기능 개발할 때 Prompter를 어떻게 쓰는지 익힙니다.

### 5-1. 평소 사용 흐름

```
1. claude 실행
2. @prompter 호출 + 거친 요청
3. Prompter 질문에 답변 (또는 바로 프롬프트 받기)
4. 받은 "복붙용 프롬프트" 복사
5. 복사한 걸 그대로 다음 메시지로 붙여넣기 → 팀 리드가 실행
6. 5-role 팀이 작업 시작!
```

### 5-2. 실전 예시 (이렇게 하면 됩니다)

**당신의 짧은 요청**:
```
@prompter

가위바위보 게임 추가하고 싶어. 컴퓨터랑 대결. Excel 모드도 지원.
```

**Prompter가 반환**:
```
## 파악한 내용
- 기존 게임들이 frontend/src/games/ 아래 각각 폴더로 분리됨
- 일반/Excel 모드 토글 로직은 기존 게임에서 패턴 확인 가능
- 랭킹 시스템 있으니 점수 관리 가능

## 가정한 내용
- 3판 2선승제 (일반적 규칙)
- 단순 랜덤 (컴퓨터 AI 없음)

## 팀 구성 추천
풀 5-role 팀

## 추가 확인 필요 사항
1. 승수를 랭킹에 기록할 건지? (단판 점수 시스템에 맞출지)
2. 컴퓨터 난이도 레벨 구분 필요한가?

답변 후 최종 프롬프트 드릴게요.

---
## 복붙용 프롬프트 (임시)
[긴 프롬프트]
```

**당신의 답변**:
```
1. 승수를 랭킹에 기록 (연승 기록)
2. 난이도 구분 없음
```

**Prompter 최종 반환** → **복사** → **다음 메시지에 붙여넣기** → **팀 시작**

### 5-3. Prompter를 언제 쓰고 언제 안 쓰나

**Prompter 쓰세요**:
- 새 기능 개발할 때
- 여러 팀원이 필요한 작업
- 내가 지금 뭘 요청해야 할지 헷갈릴 때
- 스펙이 머릿속에만 있고 정리가 안 됐을 때

**Prompter 안 써도 됨**:
- "오타 하나 고쳐줘" 같은 초간단 작업
- 문서만 수정 (README 업데이트 등)
- 이미 구체적인 프롬프트를 내가 다 썼을 때
- 팀 리드와 대화 중 팔로우업 지시 ("아까 그 부분 수정해줘")

### 5-4. 세션 구성 팁

**같은 세션에서 Prompter + 팀 운영 (단순한 경우)**:
```bash
claude
```
하나의 세션에서 Prompter 호출 → 프롬프트 받기 → 같은 세션에 붙여넣기

**별도 세션으로 분리 (추천)**:
```
터미널 1: Prompter 전용
터미널 2: 팀 리드 + 5명
```

이렇게 하면 **둘이 섞이지 않아** 깔끔합니다.
VS Code에서 터미널 두 개 열어서 각각 `claude` 실행하면 됩니다.

## 자주 겪는 문제

### 문제 1: `@prompter` 불러도 반응이 없음

**원인 A**: `.claude/agents/prompter.md`가 실제로 없거나 위치가 잘못됨

**해결**:
```bash
ls .claude/agents/prompter.md
```

파일이 없으면 1단계부터 다시.

**원인 B**: 파일 내용이 깨짐 (frontmatter 형식 오류)

**해결**: 파일 열어서 맨 위가 `---`로 시작하는지, `name:`, `description:` 등이 있는지 확인.

### 문제 2: Prompter가 계속 질문만 하고 프롬프트를 안 줌

**원인**: 요청이 너무 모호함

**해결**: 답변하면서 "이제 프롬프트를 달라"고 명시:
```
위 답변 반영해서 복붙용 프롬프트 지금 만들어줘.
```

### 문제 3: Prompter가 파일을 수정하려고 함

**원인**: `tools:` 설정이 잘못됨 (Write/Edit 섞여있음)

**해결**: `.claude/agents/prompter.md` 열어서 `tools:` 라인 확인:
```markdown
tools: Read, Glob, Grep, Bash
```

`Write`나 `Edit`이 포함돼 있으면 **지워야 합니다**. 수정 후 저장.

### 문제 4: Prompter가 준 프롬프트가 너무 장황함

**해결**: Prompter에게 피드백:
```
방금 프롬프트 너무 길어. 체크포인트 섹션을 3개 이하로 줄이고
영향 범위 섹션은 없애고 다시 만들어줘.
```

### 문제 5: Prompter가 없는 규칙을 만들어냄

**원인**: CLAUDE.md 내용 반영이 부족

**해결**: 명시적으로 지시:
```
CLAUDE.md의 금기 섹션 꼭 확인하고, 그 규칙을 프롬프트에 반영해줘.
```

### 문제 6: 같은 요청을 여러 번 다듬어야 할 때

**해결**: 이전 프롬프트 참고해서 개선 요청:
```
방금 프롬프트에서 [이 부분]만 바꿔서 다시 줘.
나머지는 그대로 유지.
```

Prompter는 이전 맥락 기억하고 있으니 쉽게 수정 가능.

### 문제 7: Prompter가 "추가 확인 필요 사항" 질문이 너무 많음

**원인**: 요청이 너무 큼 — 한 번에 여러 기능을 던짐

**해결**: 요청을 쪼개기:
```
@prompter

우선 이것만:
[작은 요청 하나]
```

## Prompter 활용 팁

### 팁 1: Prompter 전용 세션을 따로 돌리기

VS Code 터미널을 둘로 분할:
- 왼쪽: `claude` → Prompter 호출 전용
- 오른쪽: `claude` → 팀 리드 + 5명

왼쪽에서 받은 프롬프트를 오른쪽에 붙여넣는 구조가 깔끔합니다.

### 팁 2: 자주 쓰는 요청 패턴은 짧게

익숙해지면 이런 식으로 축약해도 됩니다:
```
@prompter [가위바위보 게임, Excel 모드 포함, 단판 승수 기록]
```

Prompter가 알아서 빠진 정보 질문하거나 가정해서 처리.

### 팁 3: 받은 프롬프트를 기록해두기

좋은 프롬프트는 복사해서 `docs/prompt-examples/` 같은 데 저장해두세요.
나중에 비슷한 기능 만들 때 템플릿으로 재사용 가능.

### 팁 4: Prompter가 만든 프롬프트 검토 체크리스트

받은 프롬프트를 **그대로 쓰지 말고** 한 번 훑어보세요:

- [ ] 스펙 Must/Should에 내가 원하는 거 맞나
- [ ] Excel 모드 설정이 내 의도와 일치하나
- [ ] 팀 구성이 과하거나 부족하지 않나
- [ ] 체크포인트가 합리적인가
- [ ] 금기 사항이 빠진 건 없나

이상하면 Prompter에게 수정 요청.

### 팁 5: Prompter도 진화시키기

사용하다가 "아, 이 부분은 항상 추가됐으면 좋겠다" 싶은 패턴이 생기면:

```
@prompter, 앞으로 모든 프롬프트에 [X 조건]을 기본으로 넣어줘.
네 정의 파일(.claude/agents/prompter.md)에 반영해주면 좋겠어.
```

Prompter가 자기 정의 파일 수정 제안 → 당신 승인 → 반영.
