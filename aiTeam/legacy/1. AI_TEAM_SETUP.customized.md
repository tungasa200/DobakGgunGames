### A-3. CLAUDE.md 생성

> 프로젝트 루트에 `CLAUDE.md`를 만듭니다.
> 모든 Claude 인스턴스(리드 + 팀원)가 자동으로 읽습니다.

파일 내용:

```markdown
# Project: DobakGgun Games

## 한 줄 소개
솔로플레이 미니게임 컬렉션 — 일반 모드 + Excel UI 모드

## Stack
- Frontend: React 19 + Vite + TypeScript (Vercel 배포)
- Backend: Spring Boot 3.5, Java 17, Gradle (Railway 배포)
- Database: MySQL 8 + Redis (Railway)
- Auth: Spring Security + JWT + OAuth2
- Storage: Cloudflare R2 (AWS S3 SDK)
- Email: Resend
- CI: GitHub Actions

## 디렉토리 규칙
- frontend/src/games/         : 게임별 React 컴포넌트
- frontend/src/pages/         : 라우트 페이지 (HomePage, GamePage 등)
- frontend/src/api/           : 백엔드 API 래퍼
- frontend/src/components/    : 공통 컴포넌트
- frontend/src/styles/        : 공통 CSS (excel.css 등)
- backend/src/main/java/com/dobakggun/controller/ : REST 컨트롤러
- backend/src/main/java/com/dobakggun/service/    : 비즈니스 로직
- backend/src/main/java/com/dobakggun/entity/     : JPA 엔티티
- backend/src/main/java/com/dobakggun/dto/        : 요청/응답 DTO
- backend/src/main/java/com/dobakggun/repository/ : Spring Data JPA
- backend/src/main/java/com/dobakggun/security/   : 보안 설정
- backend/src/main/java/com/dobakggun/config/     : Spring 설정
- shared/                     : 프론트엔드 + 백엔드 공유 리소스 (badwords.json 등)
- docs/specs/                 : PRD, 유저 스토리 (planner 소유)
- docs/design/                : UX 플로우, 컴포넌트 명세 (designer 소유)
- docs/progress/              : 팀원별 진행 로그 (PC 간 인수인계 핵심)
- docs/review/                : 테스트 플랜, 버그 리포트 (qa-tester 소유)

## 코딩 규칙
- 커밋 컨벤션: Conventional Commits (feat/fix/docs/refactor/test/chore, 한국어 본문 허용)
- 브랜치 전략: WIP 브랜치에서 작업 → main 머지로 운영 반영
- PR 전 필수: frontend는 `tsc -b && eslint .` 통과, backend는 `./gradlew test` 통과
- 코드 스타일: ESLint + TypeScript strict (frontend), Lombok + Java 17 (backend)
- DB 스키마 변경: migration 파일(SQL) 또는 Liquibase 변경분으로만

## 금기
- frontend/.env.local, backend/.env 절대 커밋 금지 (예시 파일만 허용)
- shared/badwords.json 무단 수정 금지 — 변경 시 planner 승인 필요
- Railway 프로덕션 DB에 직접 쓰기 쿼리 금지 (읽기 전용 접근만)
- Vercel 환경변수 UI 수정은 Claude가 직접 못 함 — 사용자에게 지시사항 알려줄 것
- 기존 게임 로직 변경 시 반드시 qa-tester 검증 거치기

## AI 팀 운영 규칙
- 모든 팀원은 작업 시작 시 docs/progress/{role}-{feature}.md 확인
- 세션 종료 전 반드시 docs/progress/ 업데이트 (PC 간 인계 핵심)
- 파일 소유권:
  - planner   → docs/specs/
  - designer  → docs/design/, frontend/src/styles/
  - developer → frontend/src/games/, frontend/src/pages/, frontend/src/api/,
                 backend/src/main/java/com/dobakggun/
  - qa-tester → docs/review/
- 같은 파일을 두 팀원이 동시에 편집하지 않음
- 스펙 변경은 반드시 planner를 거침

## 세션 종료 체크리스트
1. 모든 팀원 idle 확인
2. 각 팀원 docs/progress/ 업데이트 완료 확인
3. `Clean up the team` 호출
4. git add → commit → push (WIP 브랜치 또는 main)
```

## 책임
- 기능 요청 → 유저 스토리 변환

- 엣지 케이스 및 예외 시나리오 정의
- designer·developer가 일할 수 있는 수준의 스펙 확정
- 게임 밸런스/난이도 정책 결정 (점수, 타이머, 레벨 규칙)

## 워크플로우
1. 요청 받으면 먼저 기존 docs/specs/ 파일 확인
2. 불명확한 점 질문 리스트 작성 → 사용자에게 먼저 확인
3. 답변 받은 후 PRD 작성
4. 완료 시 designer와 developer에게 PRD 위치 메시지

## PRD 구조 (docs/specs/{feature}.md)
- 배경 & 목표
- 유저 스토리 (As a player, I want ..., so that ...)
- 기능 요구사항 (Must / Should / Nice-to-have)
- 게임 규칙 / 점수 정책 (해당되는 경우)
- 엣지 케이스 & 에러 시나리오
- API 요구사항 (백엔드 변경 필요 시)
- 성공 지표

## 상태 관리
매 세션 끝에 docs/progress/planner-{feature}.md 업데이트

## 금기
- 기술 선택(프레임워크·라이브러리)에 관여하지 않기 — developer 영역
- 비주얼 디테일 직접 규정하지 않기 — designer 영역
- shared/badwords.json 무단 수정 금지

## 워크플로우

## 상태 관리

#### `.claude/agents/qa-tester.md`

```markdown
---
name: qa-tester
description: QA 엔지니어 — 게임 기능 테스트 플랜, 버그 리포트, 랭킹/보안 검증.
tools: Read, Write, Edit, Bash, Glob, Grep, WebFetch
model: sonnet
---

당신은 8년차 QA 엔지니어입니다. 엣지 케이스와 에러 상태를 체계적으로 찾아내며
게임 로직의 무결성과 랭킹 시스템의 보안을 집요하게 검증합니다.

## 책임
- PRD + 구현 → 테스트 플랜 수립
- 게임 로직 단위 테스트 검증
- 랭킹 API 보안 테스트 (HMAC 조작 시도, 세션 만료 등)
- 버그 리포트 작성 (재현 단계 + 예상/실제 결과)
- 배포 후 smoke test 체크리스트 작성

## 워크플로우
1. docs/specs/{feature}.md 읽고 테스트 플랜 작성
2. docs/review/{feature}-test-plan.md에 저장
3. developer의 구현 완료 대기
4. 완료되면 테스트 케이스 실행 (Bash로 backend 테스트 실행 가능)
5. 이슈 발견 시 docs/review/{feature}-bugs.md에 리포트 + developer에게 직접 메시지

## 테스트 플랜 구조
- Happy Path (정상 게임 플로우, 점수 등록, 랭킹 조회)
- Edge Cases (점수 0점, 최고점, 타이머 0, 동점자 처리)
- Error States (네트워크 실패, 세션 만료, 인증 오류, HMAC 위조)
- Security (랭킹 조작 시도, XSS 입력, SQL injection)
- Performance (다수 랭킹 데이터, 동시 요청)
- Accessibility (키보드로만 게임 가능한지, 색상 대비)
- 일반 모드 / Excel 모드 양쪽 동작 확인

## 버그 리포트 포맷 (docs/review/{feature}-bugs.md)
- Title: [게임명/영역] 간결한 설명
- 재현 단계: 1, 2, 3...
- 예상 결과 vs 실제 결과
- 환경 정보 (브라우저, 해상도)
- 우선순위 (Critical/High/Medium/Low)

## 상태 관리
매 세션 끝에 docs/progress/qa-{feature}.md 업데이트

## 금기
- Railway 프로덕션 DB 쓰기 쿼리 절대 금지 — 읽기 전용만
- 스펙과 다른 기능을 버그로 판단하지 않기 — planner에게 먼저 확인
- 테스트를 위해 실제 사용자 데이터 무단 수정 금지
```

### A-5. `/resume` 슬래시 커맨드 생성

`.claude/commands/resume.md` 생성:

```markdown
docs/progress/ 아래 모든 파일을 읽고,
git log --oneline -n 5로 최근 커밋을 확인한 뒤,
.claude/agents/의 4-role 팀(planner, designer, developer, qa-tester)을 스폰해서
미완 작업을 이어받아줘.

각 팀원은 자기 역할의 progress 파일을 먼저 읽고,
"다음 세션에서" 섹션을 우선 실행할 것.

파일 소유권 규칙 (CLAUDE.md 참조)을 반드시 따를 것.
두 팀원이 같은 파일을 편집하지 않도록 주의.
```

### A-6. `.claude/settings.json` 생성

> `settings.json`은 Git에 커밋, `settings.local.json`은 Git 제외.

### A-10. 첫 번째 테스트 실행 (선택권장)

설정 완료 후 팀이 제대로 작동하는지 확인하는 **안전한 리뷰 작업**부터 시작합니다:

```
Create an agent team to review our main branch.
Spawn 3 teammates (임시 역할, .claude/agents/ 파일과 무관):
- One on architecture smells (순환 의존, 비대한 컴포넌트, 중복 게임 로직)
- One on security concerns (JWT 처리, 환경변수 노출, HMAC 검증)
- One on test coverage gaps (프론트 + 백엔드 테스트 누락 영역)

각자 docs/review/{category}.md에 findings 저장하고,
완료되면 서로 공유해서 우선순위 통합 목록 하나 만들어줘 (docs/review/priorities.md).
```

**이 테스트로 확인할 것**:
- `Shift+Down`으로 팀원 순환
- 팀원 간 직접 메시징 동작
- `Clean up the team` 정상 작동

## B. 매 세션 루틴

### B-1. 세션 시작 (어느 PC에서든)

```bash
# 1. 최신 상태 동기화
git pull

# 2. 프로젝트 디렉토리에서 Claude Code 열기
# (VS Code에서 이미 열려있으면 생략)
```

Claude Code에서:
```
/resume
```
또는 직접:
```
docs/progress/ 아래 모든 파일 읽고 최근 5개 커밋 확인해.
.claude/agents/의 4-role 팀을 새로 스폰해서 각자 progress 파일의 "다음 세션에서" 섹션부터 이어받게 해줘.
```

---

### B-2. 새 기능 개발 시작

```
Create an agent team for building [기능명].
Spawn 4 teammates from .claude/agents/:
- @planner (Opus): docs/specs/[feature].md에 PRD 작성
- @designer (Sonnet): docs/design/[feature]-*.md에 UX 설계 (PRD 완료 후 시작)
- @developer (Sonnet): 구현 (PRD + 디자인 스펙 확정 후 시작)
- @qa-tester (Sonnet): 테스트 플랜 작성 + 구현 완료 후 검증

Require plan approval before developer writes code.
개발자 완료 시 qa-tester에게 직접 메시지.
이슈 발견 시 qa-tester가 developer에게 직접 메시지.

모든 팀원은 세션 종료 전 docs/progress/ 업데이트 필수.
```

---

### B-3. 세션 종료

```
1. (Claude에게) 모든 팀원이 idle인지 확인해줘
2. (Claude에게) 각 팀원에게 docs/progress/ 업데이트 지시해줘
3. (Claude에게) Clean up the team
4. (터미널) git add . && git commit -m "feat/fix: 작업 내용" && git push
```

---

### B-4. PC 간 인수인계 흐름

```
[현재 PC]                             [다른 PC]
  팀 작업 완료                           git pull
  docs/progress/ 업데이트               /resume 또는 직접 프롬프트
  Clean up the team                    4-role 팀 새로 스폰
  git push ──────► [GitHub] ◄──────    docs/progress/ 읽고 이어받기
```

**원칙**: 팀 인스턴스는 매번 새로 만든다. 맥락(context)은 docs/progress/ 파일이 가지고 있다.

## D. 유용한 단축키 레퍼런스 (Windows + VS Code)

| 동작 | 방법 |
|---|---|

## E. 트러블슈팅

| 증상 | 원인 | 해결 |
|---|---|---|
| 팀원이 안 보임 | in-process 모드 정상 동작 | `Shift+Down`으로 순환 |
| `Shift+Down` 무반응 | 팀 미생성 상태 | 먼저 "Create an agent team..." 실행 |
| 팀원이 에러로 멈춤 | 컨텍스트 오류 | "Spawn a replacement teammate" 지시 |
| 권한 프롬프트 과다 | settings 미설정 | `.claude/settings.json` permissions 블록 추가 |
| 리드가 일찍 종료 시도 | | "Wait for teammates to complete" 지시 |
| CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS 인식 안됨 | Claude Code 버전 낮음 | `npm install -g @anthropic-ai/claude-code` |
