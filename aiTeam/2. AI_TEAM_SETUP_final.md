## 2. 배경과 전략

### 왜 이걸 하는가
- Claude **Max 플랜의 여유 토큰 예산**을 활용한 고강도 워크플로우 구축
- 단일 Claude 대화가 아니라 **여러 Claude 인스턴스가 협업**하는 병렬 작업
- 기획/디자인/프론트엔드/백엔드/QA — **5-role AI 팀**을 기존 프로젝트에 얹기
- 집 PC ↔ 회사 PC 양쪽에서 **동일한 팀 경험**으로 작업 이어가기

### 전략적 전제
- Agent Teams는 단일 세션보다 토큰을 **훨씬 많이 씀** → Max 플랜이므로 비용이 아닌 활용 기회
- 팀의 **정의(role definition)는 Git에 커밋** → PC 간 동기화 자동 해결
- 팀의 **실행 상태는 PC-local** → 아티팩트(파일) 기반 인수인계로 우회
- Git 레포를 "단일 진실 소스"로 두고 **팀은 소모품** 취급 (매번 새로 스폰)
- Developer를 **프론트엔드/백엔드 2명으로 분리** → 실제 병렬 작업 가능

### 토큰 운영 가이드라인 (Max 플랜이어도 유효)
- **Routine 작업**은 단일 Claude 세션으로 처리 (팀 안 띄움)
- **팀 스폰은** 새 기능 개발 / 복잡한 리팩토링 / PR 리뷰에만
- 팀원 3~5명이 스위트 스팟 (우리는 5명)
- 팀원당 태스크 5~6개가 적절 (너무 적으면 조율 오버헤드, 너무 많으면 컨텍스트 스위칭)
- 세션 2~3시간 이내 권장 — 그 이상이면 분할 + 커밋 + 재개

## 3. 아키텍처

### 팀 구성

```
[Team Lead] — 메인 Claude Code 세션 (당신이 직접 지시)
    │
    ├── planner            (Opus)   — PRD, 유저 스토리, 스펙 확정
    ├── designer           (Sonnet) — UX 플로우, 컴포넌트 명세
    ├── developer-frontend (Sonnet) — React 19 + TS + Vite 구현
    ├── developer-backend  (Sonnet) — Spring Boot + JPA 구현
    └── qa-tester          (Sonnet) — 테스트 플랜, E2E, 버그 리포트
```

각 팀원은 **독립된 컨텍스트 창**을 가지며, **서로 직접 메시징** 가능 (리드를 거치지 않음).
Developer 분리로 **프론트엔드 작업과 백엔드 작업이 진짜 병렬**로 돌아갑니다.

### Subagents vs Agent Teams

| | Subagents | **Agent Teams (우리 선택)** |
|---|---|---|
| 통신 | 메인 에이전트에게만 보고 | 팀원끼리 직접 메시징 |
| 컨텍스트 | 독립적, 결과만 반환 | 완전 독립 세션 |
| 토큰 비용 | 낮음 | 높음 (각 팀원이 풀 세션) |
| 적합한 작업 | 포커스된 단일 태스크 | 협업 필요한 복잡 작업 |

이 프로젝트는 **팀원 간 핸드오프**(기획→디자인→프론트/백 병렬→QA)가 필수라서 Agent Teams를 선택.

### PC 간 핸드오프 모델

```
[집 PC 세션]                              [회사 PC 세션]
 Lead + 5팀원                              Lead + 5팀원 (새로 스폰)
     │                                          ▲
     │ docs/progress/ 업데이트                  │ /resume (progress 읽고 이어받기)
     │ Clean up the team                        │ 팀 새로 스폰
     ▼                                          │
     git push ──────►  [GitHub 레포]  ◄────── git pull
                    (단일 진실 소스)
```

**원칙**: 팀 인스턴스는 매번 새로 만든다. 맥락은 `docs/progress/` 파일이 붙들고 있다.

### A-3. CLAUDE.md 생성

프로젝트 루트에 `CLAUDE.md`를 만듭니다. 모든 Claude 인스턴스(리드 + 팀원)가 자동으로 읽습니다.

```markdown
# Project: DobakGgun Games

## 한 줄 소개
솔로플레이 미니게임 컬렉션 — 일반 모드 + Excel UI 모드

## Stack
- Frontend: React 19 + Vite + TypeScript (Vercel 배포)
- Backend: Spring Boot 3.5, Java 17, Gradle (Railway 배포)
- Database: MySQL 8 + Redis (Railway)
- Auth: Spring Security + JWT + OAuth2
- Storage: Cloudflare R2 (AWS S3 SDK)
- Email: Resend
- CI: GitHub Actions

## 디렉토리 규칙
- frontend/src/games/         : 게임별 React 컴포넌트
- frontend/src/pages/         : 라우트 페이지 (HomePage, GamePage 등)
- frontend/src/api/           : 백엔드 API 래퍼
- frontend/src/components/    : 공통 컴포넌트
- frontend/src/styles/        : 공통 CSS (excel.css 등)
- backend/src/main/java/com/dobakggun/controller/ : REST 컨트롤러
- backend/src/main/java/com/dobakggun/service/    : 비즈니스 로직
- backend/src/main/java/com/dobakggun/entity/     : JPA 엔티티
- backend/src/main/java/com/dobakggun/dto/        : 요청/응답 DTO
- backend/src/main/java/com/dobakggun/repository/ : Spring Data JPA
- backend/src/main/java/com/dobakggun/security/   : 보안 설정
- backend/src/main/java/com/dobakggun/config/     : Spring 설정
- shared/                     : 프론트엔드 + 백엔드 공유 리소스 (badwords.json 등)
- docs/specs/                 : PRD, 유저 스토리 (planner 소유)
- docs/design/                : UX 플로우, 컴포넌트 명세 (designer 소유)
- docs/progress/              : 팀원별 진행 로그 (PC 간 인수인계 핵심)
- docs/review/                : 테스트 플랜, 버그 리포트 (qa-tester 소유)

## 코딩 규칙
- 커밋 컨벤션: Conventional Commits (feat/fix/docs/refactor/test/chore, 한국어 본문 허용)
- 브랜치 전략: WIP 브랜치에서 작업 → main 머지로 운영 반영
- PR 전 필수: frontend는 `tsc -b && eslint .` 통과, backend는 `./gradlew test` 통과
- 코드 스타일: ESLint + TypeScript strict (frontend), Lombok + Java 17 (backend)
- DB 스키마 변경: migration 파일(SQL) 또는 Liquibase 변경분으로만

## 환경변수 정책 (중요)
- **로컬 `.env` 파일은 템플릿/참조용이며 실제 사용되지 않음**
- **실제 환경변수는 Vercel (프론트) / Railway (백엔드) UI에만 저장됨**
- 코드에서 새 환경변수를 참조하게 되면, 팀원은 반드시 사용자에게 다음을 알림:
  - 추가해야 할 변수명
  - Vercel 또는 Railway 중 어디에 넣어야 하는지
  - 값의 형식/예시 (실제 값은 사용자가 직접 입력)
- `.env.example` 파일이 있는 경우에만 변수명 반영 (값은 절대 넣지 않음)
- `.env`, `.env.local`, `.env.production` 등 실제 값이 든 파일은 커밋 금지

## 금기
- shared/badwords.json 무단 수정 금지 — 변경 시 planner 승인 필요
- Railway 프로덕션 DB에 직접 쓰기 쿼리 금지 (읽기 전용 접근만)
- Vercel 환경변수 UI 수정은 Claude가 직접 못 함 — 사용자에게 지시사항 전달
- 기존 게임 로직 변경 시 반드시 qa-tester 검증 거치기

## Excel 모드 정책
- 이 프로젝트는 **일반 모드**와 **Excel UI 모드** 두 가지 테마를 가짐
- **Excel 모드 적용 여부는 사용자가 기능 요청 시점에 명시**
- 사용자가 "Excel 모드도 필요"라고 지시한 경우:
  - planner는 PRD "모드 적용 범위" 필드에 명시
  - designer는 양쪽 모드 명세 작성 필수
  - developer-frontend는 양쪽 구현 필수
  - qa-tester는 양쪽 동작 검증 필수 — 한쪽만 구현된 PR은 반려
- 사용자 지시가 없으면 일반 모드만 (나중에 추가 요청으로 Excel 모드 확장 가능)

## CI/CD 정책
- 팀원은 CI 실패 로그를 직접 모니터링하지 않음
- 빌드/배포 실패 시 **사용자가 로그를 긁어서 제공**
- 제공받은 로그를 기반으로 originator 팀원(developer-frontend 또는 -backend)이 진단 및 수정
- 수정 후 다시 push → 사용자가 성공/실패 결과 알림

## AI 팀 운영 규칙
- 모든 팀원은 작업 시작 시 docs/progress/{role}-{feature}.md 확인
- 세션 종료 전 반드시 docs/progress/ 업데이트 (PC 간 인계 핵심)
- 파일 소유권:
  - planner            → docs/specs/
  - designer           → docs/design/, frontend/src/styles/
  - developer-frontend → frontend/
  - developer-backend  → backend/
  - qa-tester          → docs/review/
  - shared/ 수정은 반드시 planner 승인 후 담당 developer가 수행
- 같은 파일을 두 팀원이 동시에 편집하지 않음
- 스펙 변경은 반드시 planner를 거침
- 프론트/백 양쪽이 필요한 기능은 API 계약(DTO, 엔드포인트)을 **먼저 확정** 후 병렬 작업

## 세션 종료 체크리스트
1. 모든 팀원 idle 확인
2. 각 팀원 docs/progress/ 업데이트 완료 확인
3. `Clean up the team` 호출
4. git add → commit → push (WIP 브랜치 또는 main)
```

### A-4. 에이전트 역할 파일 생성 (5개)

#### `.claude/agents/planner.md`

```markdown
---
name: planner
description: 기획자 — 게임 기능/정책 스펙 작성, 유저 스토리 정의, PRD 확정. 개발 전 명세를 완성하는 역할.
tools: Read, Write, Edit, Glob, Grep, WebFetch, WebSearch
model: opus
---

당신은 경험 많은 프로덕트 기획자입니다. DobakGgun Games의 미니게임 컬렉션에 대한 깊은 이해를 바탕으로
사용자 관점에서 기능을 정의하고 개발 가능한 수준까지 스펙을 구체화합니다.

## 책임
- 기능 요청 → 유저 스토리 변환
- PRD 작성 → docs/specs/{feature}.md
- 엣지 케이스 및 예외 시나리오 정의
- designer·developer-frontend·developer-backend가 일할 수 있는 수준의 스펙 확정
- 게임 밸런스/난이도 정책 결정 (점수, 타이머, 레벨 규칙)
- API 계약 초안 (필요 시 developer-backend와 협의해 확정)

## 워크플로우
1. 요청 받으면 먼저 기존 docs/specs/ 파일 확인
2. 불명확한 점 질문 리스트 작성 → 사용자에게 먼저 확인
3. 답변 받은 후 PRD 작성
4. 완료 시 designer, developer-frontend, developer-backend에게 PRD 위치 메시지
5. 구현 중 스펙 질문 오면 빠르게 답변

## PRD 구조 (docs/specs/{feature}.md)
- 배경 & 목표
- 유저 스토리 (As a player, I want ..., so that ...)
- **모드 적용 범위** (일반 모드 / Excel 모드 / 양쪽) — 사용자 지시 기준으로 명시
- 기능 요구사항 (Must / Should / Nice-to-have)
- 게임 규칙 / 점수 정책 (해당되는 경우)
- 엣지 케이스 & 에러 시나리오
- API 요구사항 (백엔드 변경 필요 시 초안 DTO/엔드포인트)
- 성공 지표
- 오픈 퀘스천 (답변 필요한 것)

## 상태 관리
매 세션 끝에 docs/progress/planner-{feature}.md 업데이트

## 금기
- 기술 선택(프레임워크·라이브러리)에 관여하지 않기 — developer 영역
- 비주얼 디테일 직접 규정하지 않기 — designer 영역
- shared/badwords.json 무단 수정 금지
- Excel 모드 적용 여부를 독단적으로 결정하지 않기 — 사용자 지시 따름
```

Create an agent team to review our main branch.
Spawn 3 teammates (임시 역할, .claude/agents/ 파일과 무관):
- One on architecture smells (순환 의존, 비대한 컴포넌트, 중복 게임 로직)

- `Shift+Down`으로 팀원 순환

## C. AI 팀에게 지시하는 법

### C-1. 핵심 원칙 — "리드에게, 자연어로, 구조화해서"

당신은 **대부분 Team Lead에게만** 지시합니다. 리드가 팀원에게 분배해요.
직접 팀원에게 말할 때는 `Shift+Down`으로 순환 후 특정 인스트럭션을 넣을 때뿐입니다.

### C-2. 좋은 지시 패턴 4가지

**패턴 1: 목표 + 경계 + 산출물**
```
❌ "대시보드 만들어줘"
✅ "목표: [1문장] / 경계: [제외할 것] / 산출물: PRD, 디자인 스펙, 구현, E2E 테스트"
```

**패턴 2: 한 번에 한 기능**
```
❌ "대시보드 + 알림 + 프로필 편집 다 해줘"
✅ "대시보드만 먼저. 알림/프로필은 다음 세션."
```

**패턴 3: 역할 존중**
```
❌ "planner야, 이거 React + Chart.js로 구현해"
✅ "planner에게 '기술 선택은 developer 영역, 필요한 차트 종류만 식별'이라고 전달해줘"
```

**패턴 4: 체크포인트 명시**
```
✅ "각 단계마다 내 승인 받고 진행. PRD → 디자인 → 구현 플랜 → QA 완료.
    각 체크포인트에서 팀원 대기시키고 나한테 알려."
```

### C-3. 지시 상황별 템플릿

**🟢 새 기능 개발**: B-2의 표준 템플릿 사용

**🟡 진행 중 방향 수정**:
```
Designer한테 지금 하는 거 중단하라고 해.
[이유]로 방향 바꿀 거야.
현재까지 산출물 중 재사용 가능한 부분 식별하고 대기.
```

**🔵 세션 재개**: `/resume` 사용

**🟠 특정 팀원 직접 지시** (`Shift+Down`으로 선택 후):
```
잠깐, 지금 작업 멈춰. [상세 피드백] / [대체 접근 제안]
```

### C-4. 팀원에게 직접 말해야 할 때

대부분은 리드에게 말하지만 다음 상황에선 직접:
- **깊은 피드백**: developer에게 "이 접근보다 [패턴] 쓰는 게 낫겠어. 이유는..."
- **긴급 중단**: 잘못된 방향 즉시 멈춰야 할 때
- **개별 질문**: "planner, 저 엣지 케이스 어떻게 처리할 생각?"
- **코칭**: qa-tester에게 "이 테스트 케이스 더 공격적으로"

### C-5. 실전 팁 6가지

1. **첫 지시는 길게, 이후는 짧게** — 시작 프롬프트는 구체적으로, 일단 굴러가면 짧은 피드백으로
2. **"왜"를 한 줄이라도** — "다크모드 토큰 (이유: 야간 사용자 피드백 최다)"
3. **결정 권한 명시** — "Designer에게 레이아웃 최종 결정권 위임. Developer는 의견만"
4. **"Wait for X before Y" 체인** — 순서 명확하면 리드가 조율 토큰 덜 씀
5. **30분~1시간마다 상태 점검** — "지금 각 팀원 상태 요약해줘"
6. **팀원 수 줄이기 두려워 말기** — 이미 PRD 있으면 developer + qa 2명만

### C-6. Excel 모드 지시 시

Excel 모드가 필요한 기능이면 반드시 **요청 단계에서 명시**:

```
이 기능은 Excel 모드도 필요해.
- planner: PRD "모드 적용 범위"에 "양쪽 필수" 명시
- designer: 양쪽 모드 와이어프레임 각각
- developer-frontend: 양쪽 스타일 전환 구현
- qa-tester: 양쪽 모드 동작 검증 (한쪽만 구현되면 반려)
```

누락 시 일반 모드만 구현되고 나중에 추가 작업 필요.

### C-7. 피해야 할 안티패턴

- **리드를 "또 다른 개발자"로 취급** — 리드에게 코딩 시키지 말고 조율만
- **모든 걸 승인 받게 하기** — 중요 체크포인트만 명시
- **너무 긴 세션** — 2~3시간 이내, 그 이상은 분할 + 커밋 + 재개
- **팀원 정의 수정을 대화 중에** — `.claude/agents/*.md` 수정은 팀 해산 후, 다음 팀 스폰 시 반영

## E. 실험 기능 한계 — 반드시 알아야 할 것

Agent Teams는 **실험 기능**이라 다음 제약이 있습니다. 특히 2-PC 워크플로우와 직결:

### E-1. 세션 재개 제약 (2-PC 워크플로우 영향)
- `/resume`과 `/rewind`는 **in-process 팀원을 복원 못 함**
- 세션 재개 시 리드가 존재하지 않는 팀원에게 메시지 시도 가능
- 대응: 재개 시 "새 팀원을 스폰해" 지시하거나 `/resume` 슬래시 커맨드 사용
- → **이래서 우리는 매번 팀을 새로 만드는 전제로 설계됨**

### E-2. 한 리드 = 한 팀
- 한 Claude Code 세션은 동시에 하나의 팀만 관리
- 중첩 팀 불가 (팀원이 자기 팀을 또 만들 수 없음)

### E-3. 리드 고정
- 팀 만든 세션이 수명 동안 리드 역할
- 이 세션이 죽으면 팀 재구성 필요 (Clean up → 새 팀)
- 리드 승계 불가

### E-4. 권한 설정
- 모든 팀원은 스폰 시점 리드의 권한 모드 상속
- 팀원별 개별 권한 설정은 스폰 후에만 가능 (스폰 시엔 불가)

### E-5. 플랫폼 제약
- Split-pane은 **VS Code 내장 터미널, Windows Terminal, Ghostty에서 미지원**
- 우리는 in-process 모드 사용이라 해당 없음
- `Shift+Down`으로 팀원 순환

### E-6. 태스크 상태 지연
- 팀원이 가끔 완료 체크를 깜빡함 → 의존 작업 블록
- 대응: 리드에게 "저 팀원 상태 확인하고 완료면 업데이트해줘"

## 부록 C: 팀원 수 조절 가이드

5명 팀이 기본이지만 상황에 따라 조절:

| 상황 | 권장 구성 |
|---|---|
| 순수 리뷰 작업 (코드 리팩토링 분석 등) | 리뷰어 3명 (구조/보안/테스트) |
| PRD 이미 있음, 디자인 이미 있음 | developer-frontend + developer-backend + qa-tester (3명) |
| 백엔드만 필요한 작업 (API 추가 등) | developer-backend + qa-tester (2명) |
