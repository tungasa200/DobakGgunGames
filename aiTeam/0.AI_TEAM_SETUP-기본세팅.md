# Claude Code Agent Teams 도입 가이드

> **이 문서의 목적**
> 기존 프로젝트에 Claude Code의 Agent Teams 기능과 4-role AI 팀(기획/디자인/개발/QA)을 도입하기 위한 **세팅 브리핑**입니다.
> VS Code의 Claude Code가 이 문서를 읽고, 섹션 9의 질문을 통해 프로젝트 정보를 수집한 뒤, 이 문서를 프로젝트 맞춤으로 재작성하고 실제 파일들을 생성합니다.

---

## 0. 이 문서의 사용법

VS Code에서 Claude Code를 열고 이 문서를 레포 루트에 두세요. 그다음 Claude Code에게 다음 프롬프트를 던집니다:

```
이 프로젝트 루트의 AI_TEAM_SETUP.md를 읽어줘. 그다음:
1. 섹션 9의 질문을 나한테 순서대로 물어봐서 프로젝트 정보를 수집해
2. 내 답변 + 프로젝트 파일 구조(`ls`, `git log` 등으로 파악)를 반영해
   이 문서를 AI_TEAM_SETUP.customized.md로 재작성해
3. 커스터마이즈 버전 내 승인을 받은 뒤, 섹션 4~5에 나온 파일들을 실제 생성해
4. 마지막에 섹션 7.1의 "리뷰 팀 테스트"를 같이 실행해보자
```

## 1. 목표와 배경

### 왜 이걸 하는가
- Claude **Max 플랜의 여유 토큰 예산**을 활용해 더 강력한 워크플로우 구축
- 단일 Claude 대화가 아니라 **여러 Claude 인스턴스가 협업**하는 병렬 작업
- 기획자·디자이너·개발자·QA 테스터 역할의 **4-role AI 팀**을 기존 프로젝트에 얹기
- 집 PC ↔ 회사 PC 양쪽에서 **동일한 팀 경험**으로 작업 이어가기

### 전략적 전제
- Agent Teams는 단일 세션보다 토큰을 **훨씬 많이 씀** → Max 플랜이므로 비용이 아닌 활용 기회로 해석
- 팀의 **정의(role definition)는 Git에 커밋** → PC 간 동기화 자동 해결
- 팀의 **실행 상태는 PC-local** → 아티팩트(파일) 기반 인수인계로 우회
- Git 레포를 "단일 진실 소스"로 두고 팀은 **소모품** 취급

## 2. 아키텍처 개요

### Agent Teams 구조
- **Team Lead**: 메인 Claude Code 세션. 작업 분할, 할당, 결과 통합 담당
- **Teammates**: 각자 독립된 컨텍스트 창을 가진 Claude Code 인스턴스 4명
- **공유 리소스**: Task list + Mailbox → 팀원 간 **직접 메시징** 가능 (메인 리드를 안 거침)

### Subagents vs Agent Teams — 왜 Agent Teams를 쓰는가

| | Subagents | Agent Teams |
|---|---|---|
| 통신 | 메인 에이전트에게만 보고 | 팀원끼리 직접 메시징 |
| 컨텍스트 | 독립적, 결과만 반환 | 완전 독립 세션 |
| 토큰 비용 | 낮음 | 높음 (각 팀원이 풀 세션) |
| 적합한 작업 | 포커스된 단일 태스크 | 협업 필요한 복잡 작업 |

**이 프로젝트의 선택**: Agent Teams
- 이유: 4-role 팀원들이 서로 핸드오프하며 일해야 함 (기획→디자인→개발→QA)
- Subagent 정의는 팀원으로도 재사용 가능 → `.claude/agents/` 자산이 양쪽 모두에 쓰임

### PC 간 핸드오프 모델

```
[집 PC 세션]                          [회사 PC 세션]
 Lead + 4팀원                          Lead + 4팀원 (새로 스폰)
     │                                       ▲
     │ docs/progress/ 업데이트               │ docs/progress/ 읽고 이어받기
     │ Clean up the team                    │ 4-role 팀 새로 스폰
     ▼                                       │
     git push ───────►  [Git 레포]  ◄───── git pull
                    (단일 진실 소스)
```

**원칙**: 팀은 매번 새로 만든다. 맥락은 파일이 붙들고 있다.

## 3. 전제조건 체크

```bash
# Claude Code 버전 확인 (v2.1.32 이상 필수)
claude --version

# 업데이트/설치
npm install -g @anthropic-ai/claude-code

# 로그인 (Max 플랜 계정)
claude login

# (선택) Split-pane 원하면 tmux 또는 iTerm2 + it2 CLI
# macOS: brew install tmux
```

**OS별 권장 터미널**
- macOS: iTerm2 + `tmux -CC` 모드
- Linux: 기본 터미널 + tmux
- Windows: WSL2 + tmux (네이티브 Windows Terminal은 split-pane 미지원)

**split-pane 지원되지 않는 환경**: VS Code 내장 터미널, Windows Terminal, Ghostty
→ 이런 환경에서는 자동으로 `in-process` 모드 사용 (`Shift+Down`으로 팀원 순환)

## 4. 프로젝트 구조 세팅

### 4.1 생성할 파일/디렉토리

```
<프로젝트 루트>/
├── CLAUDE.md                       # 프로젝트 공통 규칙 (모든 팀원이 자동 읽음)
├── .claude/
│   ├── agents/
│   │   ├── planner.md              # 기획자 역할 정의
│   │   ├── designer.md             # 디자이너 역할 정의
│   │   ├── developer.md            # 개발자 역할 정의
│   │   └── qa-tester.md            # QA 테스터 역할 정의
│   ├── commands/
│   │   └── resume.md               # /resume 슬래시 커맨드
│   └── settings.json               # Agent Teams 활성화 플래그
├── docs/
│   ├── specs/                      # 기획자 산출물 (PRD, 유저 스토리)
│   ├── design/                     # 디자이너 산출물 (UX 플로우, 와이어프레임)
│   ├── progress/                   # 역할별 진행 로그 (인수인계 핵심)
│   └── review/                     # 리뷰/QA 산출물
└── .gitignore                      # .claude/settings.local.json 등 제외
```

```markdown
1. 모든 팀원 idle 확인
2. docs/progress/ 최신화 확인

4. git add → commit → push
```

## 5. 역할별 서브에이전트 템플릿

각 템플릿은 **스켈레톤**입니다. Claude Code가 프로젝트 정보를 받은 후 `[BRACKETS]` 부분과 워크플로우 세부사항을 프로젝트에 맞게 채웁니다.

### 5.1 .claude/agents/planner.md

```markdown
---
name: planner
description: 기획자 — 유저 스토리 정의, PRD 작성, 스펙 확정. 기능 요청을 개발 가능한 수준까지 구체화.
tools: Read, Write, Edit, Glob, Grep, WebFetch, WebSearch
model: opus
---

당신은 경험 많은 프로덕트 기획자입니다. [PROJECT_DOMAIN]에 대한 이해를 바탕으로
사용자 관점에서 기능을 정의하고 개발 가능한 수준까지 스펙을 구체화합니다.

## 책임
- 기능 요청 → 유저 스토리 변환
- PRD 작성 → docs/specs/{feature}.md
- 엣지 케이스 및 예외 시나리오 정의
- 디자이너/개발자가 일할 수 있는 수준의 스펙 확정

## 워크플로우
1. 요청 받으면 먼저 **질문 리스트**부터 — 불명확한 점 명확히
2. 답변 받은 후 PRD 작성
3. 완료 시 designer와 developer에게 PRD 위치 메시지
4. 구현 중 스펙 질문 오면 빠르게 답변

## PRD 구조 (docs/specs/{feature}.md)
- 배경 & 목표
- 유저 스토리 (As a [user], I want [action], so that [benefit])
- 기능 요구사항 (Must / Should / Nice-to-have)
- 엣지 케이스 & 에러 시나리오
- 성공 지표
- 오픈 퀘스천 (답변 필요한 것)

## 상태 관리
매 세션 끝에 docs/progress/planner-{feature}.md 업데이트:
- 완료한 스펙
- 진행 중인 것
- 답변 대기 중인 질문
- 다음 세션에서 할 것

## 금기
- 기술 선택(framework 등)에 지나치게 관여하지 않기 — developer 영역
- 비주얼 디테일 규정하지 않기 — designer 영역
- [PROJECT_SPECIFIC_FORBIDDEN]
```

## 6. PC 간 인수인계 워크플로우

### 6.1 docs/progress/ 파일 포맷

각 팀원이 세션 끝에 업데이트하는 형식:


## 7. 작업 시작 가이드

### 7.1 첫 테스트 — 리뷰 팀 (안전함, 파일 충돌 없음)

새 기능 개발로 곧장 시작하지 마세요. **리뷰 작업부터** 해서 팀이 어떻게 움직이는지 보세요.

```
Create an agent team to review our main branch.
Spawn 3 teammates (반드시 .claude/agents/ 외의 임시 역할로):
- One on architecture smells (순환 의존, 거대 컴포넌트, 중복)
- One on security concerns (인증 플로우, 환경변수 처리, 입력 검증)
- One on test coverage gaps

각자 docs/review/{category}.md에 findings 저장하고,
완료되면 서로 공유해서 우선순위 통합 목록 하나 만들어줘 (docs/review/priorities.md).
```

**이 테스트로 확인할 것**:
- `Shift+Down`으로 팀원 순환
- 팀원 간 직접 메시징 동작
- `Clean up the team` 정상 작동

### 7.2 풀 4-role 팀 — 새 기능 개발

리뷰 팀이 잘 돌아갔다면 이제:

```
Create an agent team for building [FEATURE_NAME].
Spawn 4 teammates from .claude/agents/:
- @planner (Opus): docs/specs/[feature].md에 PRD 작성
- @designer (Sonnet): docs/design/[feature]-*.md에 UX 설계 (PRD 완료 후 시작)
- @developer (Sonnet): 구현 (PRD + 디자인 스펙 승인 후 시작)
- @qa-tester (Sonnet): 테스트 플랜 작성 + 구현 완료 후 E2E 스크립트

Require plan approval before developer writes code.
개발자 완료 시 qa-tester에게 직접 메시지.
이슈 발견 시 qa-tester가 developer에게 직접 메시지.

모든 팀원은 세션 종료 전 docs/progress/ 업데이트 필수.
```

### 7.3 Git worktree 활용 (고급, 선택사항)

팀을 별도 워크트리에서 돌리면 본인 작업 브랜치와 안 섞입니다:

```bash

cd ../myproject-ai-team
```

## 8. 실험 기능 한계와 주의사항

### 반드시 알아야 할 한계

- `/resume`과 `/rewind`가 in-process 팀원을 **복원 못함** → 세션 재개 시 리드가 존재하지 않는 팀원에게 메시지 시도 가능. 대응: "새 팀원 스폰해" 지시 또는 `/resume` 커맨드 사용
- **한 리드가 동시에 팀 하나만** 관리 가능 (중첩 팀 불가)
- **리드 고정** — 팀 만든 세션이 수명 동안 리드. 이 세션 죽으면 팀 재구성 필요
- **권한 설정은 스폰 시점 리드 기준**으로 고정
- Split-pane은 VS Code 내장 터미널, Windows Terminal, Ghostty에서 **미지원**

### 토큰 비용 관리 (Max 플랜이어도 낭비는 방지)

- **Routine 작업**은 단일 세션 또는 서브에이전트로 해결
- **팀은 아껴 쓰기**: PR 리뷰, 새 기능 개발, 복잡한 디버깅
- **팀원 수**: 3-5명이 스위트 스팟. 5명 초과하면 조율 오버헤드가 이득 상쇄
- **팀원당 작업**: 5-6개가 적절. 너무 적으면 조율 오버헤드, 너무 많으면 컨텍스트 스위칭

### 파일 충돌 방지

두 팀원이 같은 파일을 편집하면 덮어쓰기 발생. CLAUDE.md의 "파일 소유권" 규칙이 이걸 예방합니다.

## 9. Claude Code에게 — 커스터마이즈 지시사항

> **이 섹션은 이 문서를 읽고 있는 Claude Code (VS Code에 설치된 인스턴스)를 위한 지시사항입니다.**

### 9.1 정보 수집 — 사용자에게 질문할 것

**질문 1: 프로젝트 기본**
- 프로젝트 이름은?
- 한 줄 요약?
- 현재 진행 단계? (초기 설계 / 활발한 개발 / 유지보수 / 리팩토링 중)
- 주요 유저/고객은? (내부 도구 / B2C / B2B / 오픈소스)

**질문 2: 기술 스택**
- 프론트엔드 프레임워크/라이브러리?
- 백엔드 / API 레이어?
- 데이터베이스?
- 테스트 도구? (unit + E2E 각각)
- 배포 방식?
- 그 외 중요한 라이브러리/도구?

**질문 3: 프로젝트 구조** (실제 확인)
- `ls` 또는 `tree -L 2 -I 'node_modules|.git'`로 실제 구조 파악
- `package.json` / `pyproject.toml` / 기타 설정 파일 확인
- `.gitignore` 확인
- 기존 README.md / CONTRIBUTING.md가 있으면 읽고 규칙 추출

**질문 4: 코딩 규칙**
- 커밋 컨벤션? (Conventional Commits / 자유 / 기타)
- 브랜치 전략? (main/develop/feature / GitHub Flow / 기타)
- PR 전 필수 체크? (test / lint / type-check / 빌드)
- 코드 스타일 가이드? (ESLint/Prettier 설정 있는지)

**질문 5: 금기/제약**
- 건드리면 안 되는 파일/디렉토리?
- DB 스키마 변경 정책?
- 배포/프로덕션 관련 제약?
- 환경변수 / 시크릿 관련 주의사항?

**질문 6: 팀 구성 커스터마이즈**
- 4개 역할 중 프로젝트 특성상 가중치가 다를까요? 예:
  - 백엔드 중심 → developer 역할을 backend/frontend로 분리?
  - API 프로젝트 → designer 역할이 API designer(스키마/OpenAPI)로 변경?
  - 디자인 시스템 프로젝트 → designer 비중 증가?
- MCP 서버 연결 필요? (Playwright, Supabase, GitHub, Linear 등)
- 모델 전략: planner는 Opus / 나머지 Sonnet이 기본. 다르게 원하면?

**질문 7: PC 환경**
- 주로 작업하는 OS? (macOS / Linux / WSL / Windows)
- 터미널? (iTerm2 / VS Code 내장 / Ghostty / 기타)
- Split-pane 시도할지? (아니면 in-process로 시작)

### 9.2 수집 후 생성할 파일

질문 답변 + 프로젝트 파악 정보를 바탕으로 다음 파일들을 **실제 생성**합니다:

1. `CLAUDE.md` — 4.2 스켈레톤 + 수집 정보 반영
2. `.claude/agents/planner.md` — 5.1 템플릿 + 프로젝트 맞춤
3. `.claude/agents/designer.md` — 5.2 템플릿 + 프로젝트 맞춤
4. `.claude/agents/developer.md` — 5.3 템플릿 + 프로젝트 맞춤
5. `.claude/agents/qa-tester.md` — 5.4 템플릿 + 프로젝트 맞춤
6. `.claude/commands/resume.md` — 6.4 내용
7. `.claude/settings.json` — 4.3 내용 (이미 존재하면 env 블록만 추가)
8. `.gitignore` — 4.4 항목 추가 (기존 .gitignore에 append)
9. `docs/specs/.gitkeep`, `docs/design/.gitkeep`, `docs/progress/.gitkeep`, `docs/review/.gitkeep`
10. `AI_TEAM_SETUP.customized.md` — 이 문서의 프로젝트 맞춤 버전

### 9.3 생성 완료 후

- 생성한 파일 목록을 사용자에게 diff 형태로 보여주고 승인 요청
- 승인 후 `git add` 제안 (커밋은 사용자가 직접)
- 7.1의 "리뷰 팀 테스트"를 함께 실행 제안
- 첫 팀 실행 시 사용자가 다음을 실습하도록 안내:
  - `Shift+Down`으로 팀원 순환
  - 팀원에게 직접 메시지
  - `Clean up the team`
  - `git commit` 후 `/resume`으로 재개

### 9.4 대화 시 주의

- 사용자가 Max 플랜 사용자임을 인지 — 토큰 절약보다 **활용 극대화**가 목표
- 질문을 한번에 다 던지지 말고 **섹션별로** (프로젝트 기본 → 스택 → 구조 → 규칙 → 금기 → 팀 → PC)
- 사용자가 "모르겠다 / 그냥 추천해줘"라고 하면 **합리적 기본값**으로 진행 후 추후 수정 가능함을 알림
- 프로젝트 구조 파악은 **질문하지 말고 직접 `ls`/`cat`으로 확인**
- 기존 파일을 덮어쓸 때는 반드시 먼저 보여주고 확인받기 (특히 `.gitignore`, `CLAUDE.md`)

## 부록 A: 유용한 커맨드 레퍼런스

### 팀 운영
- 팀 생성: 자연어로 "Create an agent team..."
- 팀원에게 직접 메시지: `Shift+Down`으로 순환 후 타이핑
- 팀원 세션 진입: 해당 팀원 선택 후 `Enter`
- 현재 팀원 턴 중단: `Esc`
- 태스크 리스트 토글: `Ctrl+T`
- 팀 정리: `Clean up the team`

### 디스플레이 모드 전환
```bash
# 일회성 in-process 강제
claude --teammate-mode in-process

# 영구 설정 (~/.claude.json)
{ "teammateMode": "in-process" }  # 또는 "tmux" 또는 "auto"
```

### 훅 활용 (고급)
- `TeammateIdle`: 팀원 일 멈추려 할 때 — exit 2로 피드백 주고 계속 시키기
- `TaskCreated`: 작업 생성 시 검증 (예: 스펙 링크 없으면 차단)
- `TaskCompleted`: 완료 마킹 시 검증 (예: 테스트 통과 안 하면 완료 불가)

## 부록 B: 트러블슈팅

**팀원이 안 보인다**
- In-process 모드면 `Shift+Down`으로 순환 (이미 실행 중일 수 있음)
- Split-pane 원했으면 tmux 설치 확인: `which tmux`

**권한 프롬프트가 너무 많다**
- `.claude/settings.json`의 permissions에 자주 쓰는 동작 미리 허용

**팀원이 에러로 멈춘다**
- 해당 팀원 세션 진입해서 직접 지시
- 또는 "Spawn a replacement teammate"로 교체

**리드가 작업 안 끝났는데 멋대로 종료하려 한다**
- "Wait for teammates to complete their tasks before proceeding"

**tmux 세션이 남아있다**
```bash
tmux ls
tmux kill-session -t <session-name>
```
